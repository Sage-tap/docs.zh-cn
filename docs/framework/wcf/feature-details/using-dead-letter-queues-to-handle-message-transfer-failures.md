 <span data-ttu-id="d4b26-131">从死信队列中读取消息的应用程序类似于从应用程序队列读取消息的 WCF 服务，但存在以下细微差异：</span><span class="sxs-lookup"><span data-stu-id="d4b26-131">An application that reads messages out of a dead-letter queue is similar to a WCF service that reads from an application queue, except for the following minor differences:</span></span>  
  
- <span data-ttu-id="d4b26-132">若要从系统的事务性死信队列中读取消息，统一资源标识符 (URI) 必须为以下形式：net.msmq://localhost/system$;DeadXact。</span><span class="sxs-lookup"><span data-stu-id="d4b26-132">To read messages from a system transactional dead-letter queue, the Uniform Resource Identifier (URI) must be of the form: net.msmq://localhost/system$;DeadXact.</span></span>

 <span data-ttu-id="d4b26-135">有关如何对队列进行寻址的详细信息，请参阅 [服务终结点和队列寻址](service-endpoints-and-queue-addressing.md)。</span><span class="sxs-lookup"><span data-stu-id="d4b26-135">For more information about how to address queues, see [Service Endpoints and Queue Addressing](service-endpoints-and-queue-addressing.md).</span></span>  
  
 <span data-ttu-id="d4b26-136">接收方上的 WCF 堆栈会将服务正在侦听的地址与消息上的地址相匹配。</span><span class="sxs-lookup"><span data-stu-id="d4b26-136">The WCF stack on the receiver matches addresses that the service is listening on with the address on the message.</span></span> <span data-ttu-id="d4b26-137">如果地址匹配，则调度该消息；否则，不调度该消息。</span><span class="sxs-lookup"><span data-stu-id="d4b26-137">If the addresses match, the message is dispatched; if not, the message is not dispatched.</span></span> <span data-ttu-id="d4b26-138">在从死信队列中进行读取时，这种操作可能会引发问题，原因是死信队列中的消息通常将寻址到该服务而非死信队列服务。</span><span class="sxs-lookup"><span data-stu-id="d4b26-138">This can cause problems when reading from the dead-letter queue, because messages in the dead-letter queue are typically addressed to the service and not the dead-letter queue service.</span></span> <span data-ttu-id="d4b26-139">因此，从死信队列中进行读取的服务必须安装一个地址筛选器 `ServiceBehavior`，指示堆栈独立于被寻地址来与队列中的所有消息匹配。</span><span class="sxs-lookup"><span data-stu-id="d4b26-139">Therefore, the service reading from the dead-letter queue must install an address filter `ServiceBehavior` that instructs the stack to match all messages in the queue independently of the addressee.</span></span> <span data-ttu-id="d4b26-140">具体而言，您必须将一个带有 `ServiceBehavior` 参数的 <xref:System.ServiceModel.AddressFilterMode.Any> 添加到从死信队列中读取消息的服务中。</span><span class="sxs-lookup"><span data-stu-id="d4b26-140">Specifically, you must add a `ServiceBehavior` with the <xref:System.ServiceModel.AddressFilterMode.Any> parameter to the service reading messages from the dead-letter queue.</span></span>

 <span data-ttu-id="d4b26-145">在将 `ReceiveErrorHandling` 设置为 `Reject` 以及从自定义死信队列中进行读取时，病毒消息将放入系统的死信队列。</span><span class="sxs-lookup"><span data-stu-id="d4b26-145">When `ReceiveErrorHandling` is set to `Reject`, when reading from the custom dead letter queue, the poison message is put in the system dead-letter queue.</span></span> <span data-ttu-id="d4b26-146">如果从系统的死信队列中进行读取，则将丢弃（清除）该消息。</span><span class="sxs-lookup"><span data-stu-id="d4b26-146">If reading from the system dead-letter queue, the message is dropped (purged).</span></span> <span data-ttu-id="d4b26-147">在 MSMQ 的系统死信队列中执行拒绝将丢弃（清除）该消息。</span><span class="sxs-lookup"><span data-stu-id="d4b26-147">A reject from a system dead-letter queue in MSMQ drops (purges) the message.</span></span>
