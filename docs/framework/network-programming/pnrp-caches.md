# <a name="pnrp-caches"></a><span data-ttu-id="e67c0-103">PNRP 缓存</span><span class="sxs-lookup"><span data-stu-id="e67c0-103">PNRP Caches</span></span>

<span data-ttu-id="e67c0-104">对等名称解析协议 (PNRP) 缓存是对等机上维护的算法选择的对等终结点的本地集合。</span><span class="sxs-lookup"><span data-stu-id="e67c0-104">Peer Name Resolution Protocol (PNRP) caches are local collections of algorithmically selected peer endpoints maintained on the peer.</span></span>  
  
## <a name="pnrp-cache-initialization"></a><span data-ttu-id="e67c0-105">PNRP 缓存初始化</span><span class="sxs-lookup"><span data-stu-id="e67c0-105">PNRP Cache Initialization</span></span>  

 <span data-ttu-id="e67c0-106">若要初始化 PNRP 缓存或对等名称记录集合，当对等节点启动时节点可以使用以下方法：</span><span class="sxs-lookup"><span data-stu-id="e67c0-106">To initialize the PNRP cache, or Peer Name Record Collection, when a peer node starts up, a node can use the following methods:</span></span>  
  
- <span data-ttu-id="e67c0-107">从硬盘存储中加载节点关闭时出现的持久性缓存条目。</span><span class="sxs-lookup"><span data-stu-id="e67c0-107">Persistent cache entries that were present when the node was shut down are loaded from hard disk storage.</span></span>  
  
- <span data-ttu-id="e67c0-108">如果应用程序使用 P2P 协作基础结构，将在该节点的联系人管理器中提供协作信息。</span><span class="sxs-lookup"><span data-stu-id="e67c0-108">If an application uses the P2P collaboration infrastructure, collaboration information is available in the Contact Manager for that node.</span></span>  
  
## <a name="scaling-peer-name-resolution-with-a-multi-level-cache"></a><span data-ttu-id="e67c0-109">使用多级别缓存缩放对等名称解析</span><span class="sxs-lookup"><span data-stu-id="e67c0-109">Scaling Peer Name Resolution with a Multi-Level Cache</span></span>  

 <span data-ttu-id="e67c0-110">若要保持较小的 PNRP 缓存大小，对等节点将使用多级别缓存，其中每个级别包含最大数目的条目。</span><span class="sxs-lookup"><span data-stu-id="e67c0-110">To keep the sizes of the PNRP caches small, peer nodes use a multi-level cache, in which each level contains a maximum number of entries.</span></span> <span data-ttu-id="e67c0-111">缓存中的每个级别代表 PNRP ID 数值空间 (2<sup>256</sup>) 的十分之一较小部分。</span><span class="sxs-lookup"><span data-stu-id="e67c0-111">Each level in the cache represents a one tenth smaller portion of the PNRP ID number space (2<sup>256</sup>).</span></span> <span data-ttu-id="e67c0-112">缓存中最低的级别包含本地注册的 PNRP ID 和其他数字上相近的 PNRP ID。</span><span class="sxs-lookup"><span data-stu-id="e67c0-112">The lowest level in the cache contains a locally registered PNRP ID and other PNRP IDs that are numerically close to it.</span></span> <span data-ttu-id="e67c0-113">当缓存级别填充了多达 20 个条目时，会创建新的较低级别。</span><span class="sxs-lookup"><span data-stu-id="e67c0-113">As a level of the cache is filled with a maximum of 20 entries, a new lower level is created.</span></span> <span data-ttu-id="e67c0-114">缓存中级别的最大数目近似于 log10(云中 PNRP ID 的总数)。</span><span class="sxs-lookup"><span data-stu-id="e67c0-114">The maximum number of levels in the cache is on the order of log10(Total number of PNRP IDs in the cloud).</span></span> <span data-ttu-id="e67c0-115">例如，对于有 1 亿 PNRP ID 的全局云，存在不超过 8 (=log10(100,000,000)) 个的缓存级别，以及相似数量的跃点以在名称解析期间解析 PNRP ID。</span><span class="sxs-lookup"><span data-stu-id="e67c0-115">For example, for a global cloud with 100 million PNRP IDs, there are no more than 8 (=log10(100,000,000)) levels in the cache and a similar number of hops to resolve a PNRP ID during name resolution.</span></span> <span data-ttu-id="e67c0-116">这一机制允许分布式哈希表，可为此哈希表通过以下方式解析任意 PNRP ID：将 PNRP 请求消息转发到下个最近的对等机，直到找到具有相应 CPA 的对等机。</span><span class="sxs-lookup"><span data-stu-id="e67c0-116">This mechanism allows for a distributed hash table for which an arbitrary PNRP ID can be resolved by forwarding PNRP Request messages to the next-closest peer until the peer with the corresponding CPA is found.</span></span>  
  
 <span data-ttu-id="e67c0-117">若要确保解析的完成，每次节点将条目添加到其缓存的最低级别时，都将条目的副本传到缓存的最后级别中的所有节点。</span><span class="sxs-lookup"><span data-stu-id="e67c0-117">To ensure that resolution can complete, each time a node adds an entry to the lowest level of its cache, it floods a copy of the entry to all the nodes within the last level of the cache.</span></span>  
  
 <span data-ttu-id="e67c0-118">缓存条目随着时间推移进行刷新。</span><span class="sxs-lookup"><span data-stu-id="e67c0-118">The cache entries are refreshed over time.</span></span> <span data-ttu-id="e67c0-119">陈旧条目会从缓存中移除。</span><span class="sxs-lookup"><span data-stu-id="e67c0-119">Cache entries that are stale are removed from the cache.</span></span> <span data-ttu-id="e67c0-120">结果是 PNRP ID 的分布式哈希表基于活动的终结点，不像 DNS 中地址记录和 DNS 协议不保障与地址相关的节点活跃在网络上。</span><span class="sxs-lookup"><span data-stu-id="e67c0-120">The result is that the distributed hash table of PNRP IDs is based on active endpoints, unlike DNS in which address records and the DNS protocol provide no guarantee that the node associated with the address is actively on the network.</span></span>  
  
## <a name="other-pnrp-caches"></a><span data-ttu-id="e67c0-121">其他 PNRP 缓存</span><span class="sxs-lookup"><span data-stu-id="e67c0-121">Other PNRP Caches</span></span>  

 <span data-ttu-id="e67c0-122">另一个永久性数据存储是本地缓存。</span><span class="sxs-lookup"><span data-stu-id="e67c0-122">Another persistent data store is the local cache.</span></span>  <span data-ttu-id="e67c0-123">除了 PNRP 活动需要的其他对象，它可包括与 PNRP 云或协作会话相关联的记录，这些记录在云的所有成员间安全地进行发布和同步。</span><span class="sxs-lookup"><span data-stu-id="e67c0-123">In addition to the other objects needed for PNRP activity, it may include the records associated with a PNRP cloud or collaboration session that is securely published and synchronized between all the members of the cloud.</span></span> <span data-ttu-id="e67c0-124">这一复制的存储代表了组数据的视图，它应在所有组成员间保持相同。</span><span class="sxs-lookup"><span data-stu-id="e67c0-124">This replicated store represents the view of the group data, which should be the same for all group members.</span></span> <span data-ttu-id="e67c0-125">从技术上来讲，这些对象本质上不是记录，而是去往本地缓存的应用程序、状态和对象数据。</span><span class="sxs-lookup"><span data-stu-id="e67c0-125">Technically, these objects are not records per se, but rather application, presence, and object data destined for a local cache.</span></span> <span data-ttu-id="e67c0-126">使用 PNRP 云确保对象传播到协作会话或 PNRP 云中的所有节点。</span><span class="sxs-lookup"><span data-stu-id="e67c0-126">Use of the PNRP cloud ensures that objects are propagated to all nodes in the collaboration session or PNRP cloud.</span></span>  <span data-ttu-id="e67c0-127">云成员间的记录副本使用 SSL 提供加密和数据完整性。</span><span class="sxs-lookup"><span data-stu-id="e67c0-127">Record replication between cloud members uses SSL to provide encryption and data integrity.</span></span>  
  
 <span data-ttu-id="e67c0-128">当对等机加入云时，它们不会自动接收来自其附加于的主机对等机的本地缓存数据，它们必须订阅主机对等机以接收应用程序、状态和对象数据中的更新。</span><span class="sxs-lookup"><span data-stu-id="e67c0-128">When a peer joins a cloud, they do not automatically receive local cache data from the host peer to which they attach; they have to subscribe to the host peer to receive updates in application, presence, and object data.</span></span> <span data-ttu-id="e67c0-129">初始同步后，对等机定期重新同步其复制的存储，确保所有组成员始终有相同的视图。</span><span class="sxs-lookup"><span data-stu-id="e67c0-129">After the initial synchronization, peers periodically resynchronize their replicated stores to ensure that all group members consistently have the same view.</span></span>  <span data-ttu-id="e67c0-130">协作会话或协作会话中的应用程序也可执行相同的功能。</span><span class="sxs-lookup"><span data-stu-id="e67c0-130">The collaboration session or applications within the collaboration session may also perform the same function.</span></span>  
  
 <span data-ttu-id="e67c0-131">协作会话为云启动后，应用程序可以使用云范围定义的安全性注册对等机并开始发布其信息。</span><span class="sxs-lookup"><span data-stu-id="e67c0-131">After a collaboration session has begun for a cloud, applications can register peers and begin publishing their information using the security defined by the cloud scope.</span></span> <span data-ttu-id="e67c0-132">对等机加入云时，云的安全性机制会应用到对等机，为其提供要加入的范围。</span><span class="sxs-lookup"><span data-stu-id="e67c0-132">When a peer joins a cloud, the security mechanisms for the cloud are applied to the peer, giving it a scope in which to participate.</span></span>  <span data-ttu-id="e67c0-133">然后它的记录可以在云的范围内安全发布。</span><span class="sxs-lookup"><span data-stu-id="e67c0-133">Its records can then be published securely within the scope of the cloud.</span></span> <span data-ttu-id="e67c0-134">请注意云范围可能和协作应用程序范围不相同。</span><span class="sxs-lookup"><span data-stu-id="e67c0-134">Note that cloud scope may not be the same as collaboration application scope.</span></span>  
  
 <span data-ttu-id="e67c0-135">对等机可以登记从其他对等机接收对象的偏好。</span><span class="sxs-lookup"><span data-stu-id="e67c0-135">Peers can register interest in receiving objects from other peers.</span></span> <span data-ttu-id="e67c0-136">当对象更新时，协作应用程序收到通知，新的对象传递给应用程序的所有订阅者。</span><span class="sxs-lookup"><span data-stu-id="e67c0-136">When an object is updated, the collaboration application is notified and the new object is passed to all subscribers of the application.</span></span> <span data-ttu-id="e67c0-137">例如，组聊天应用程序中的对等程序可以登记接收应用程序信息的偏好，这会向其发送所有聊天记录作为应用程序数据。</span><span class="sxs-lookup"><span data-stu-id="e67c0-137">For example, a peer in a group chat application can register interest in receiving application information, which will send it all chat records as application data.</span></span>  <span data-ttu-id="e67c0-138">这使得它可以监视云内的聊天活动。</span><span class="sxs-lookup"><span data-stu-id="e67c0-138">This allows it to monitor chat activity within the cloud.</span></span>
