## <a name="overview"></a><span data-ttu-id="264f6-111">概述</span><span class="sxs-lookup"><span data-stu-id="264f6-111">Overview</span></span>  

 <span data-ttu-id="264f6-112">集成式 Windows 身份验证的设计能使某些凭据质询响应变得通用，这意味着可以重新使用或转发这些响应。</span><span class="sxs-lookup"><span data-stu-id="264f6-112">The design of integrated Windows authentication allows for some credential challenge responses to be universal, meaning they can be re-used or forwarded.</span></span> <span data-ttu-id="264f6-113">质询响应至少应与目标特定的信息一起构造，最好还包含一些通道特定的信息。</span><span class="sxs-lookup"><span data-stu-id="264f6-113">The challenge responses should be constructed at a minimum with target specific information and preferably also with some channel specific information.</span></span> <span data-ttu-id="264f6-114">然后，服务可提供扩展保护，确保凭据质询响应包含服务主体名称 (SPN) 等服务特定的信息。</span><span class="sxs-lookup"><span data-stu-id="264f6-114">Services can then provide extended protection to ensure that credential challenge responses contain service specific information such as a Service Principal Name (SPN).</span></span> <span data-ttu-id="264f6-115">凭据交换中含有此信息时，服务就能更好防范恶意使用那些可能已被不当使用的凭据质询响应。</span><span class="sxs-lookup"><span data-stu-id="264f6-115">With this information in the credential exchanges, services are able to better protect against malicious use of credential challenge responses that might have been improperly used.</span></span>  
  
 <span data-ttu-id="264f6-116">扩展保护设计是对身份验证协议的增强，旨在减轻身份验证中继攻击。</span><span class="sxs-lookup"><span data-stu-id="264f6-116">The extended protection design is an enhancement to authentication protocols designed to mitigate authentication relay attacks.</span></span> <span data-ttu-id="264f6-117">它围绕通道和服务绑定信息的概念展开。</span><span class="sxs-lookup"><span data-stu-id="264f6-117">It revolves around the concept of channel and service binding information.</span></span>  
  
 <span data-ttu-id="264f6-118">总体目标如下所示：</span><span class="sxs-lookup"><span data-stu-id="264f6-118">The overall objectives are the following:</span></span>  
  
1. <span data-ttu-id="264f6-119">如果客户端更新为支持扩展保护，应用程序应为所有受支持身份验证协议提供通道绑定和服务绑定信息。</span><span class="sxs-lookup"><span data-stu-id="264f6-119">If the client is updated to support the extended protection, applications should supply a channel binding and service binding information to all supported authentication protocols.</span></span> <span data-ttu-id="264f6-120">仅当有要绑定到的通道 (TLS) 时，才提供通道绑定信息。</span><span class="sxs-lookup"><span data-stu-id="264f6-120">Channel binding information can only be supplied when there is a channel (TLS) to bind to.</span></span> <span data-ttu-id="264f6-121">应始终提供服务绑定信息。</span><span class="sxs-lookup"><span data-stu-id="264f6-121">Service binding information should always be supplied.</span></span>  
  
2. <span data-ttu-id="264f6-122">适当配置的已更新服务器可能会在通道和服务绑定信息出现在客户端身份验证令牌中时验证这些信息，如果通道绑定不匹配则拒绝身份验证尝试。</span><span class="sxs-lookup"><span data-stu-id="264f6-122">Updated servers which are properly configured may verify the channel and service binding information when it is present in the client authentication token and reject the authentication attempt if the channel bindings do not match.</span></span> <span data-ttu-id="264f6-123">根据部署方案，服务器可能会验证通道绑定或服务绑定或同时验证两者。</span><span class="sxs-lookup"><span data-stu-id="264f6-123">Depending on the deployment scenario, servers may verify channel binding, service binding or both.</span></span>  
  
3. <span data-ttu-id="264f6-124">更新的服务器可以基于策略接受或拒绝不包含通道绑定信息的下层客户端请求。</span><span class="sxs-lookup"><span data-stu-id="264f6-124">Updated servers have the ability to accept or reject down-level client requests that do not contain the channel binding information based on policy.</span></span>  
  
 <span data-ttu-id="264f6-125">扩展保护使用的信息由以下两个部分或其中一个部分组成：</span><span class="sxs-lookup"><span data-stu-id="264f6-125">Information used by extended protection consists of one or both of the following two parts:</span></span>  
  
1. <span data-ttu-id="264f6-126">通道绑定令牌（即 CBT）。</span><span class="sxs-lookup"><span data-stu-id="264f6-126">A Channel Binding Token or CBT.</span></span>  
  
2. <span data-ttu-id="264f6-127">服务主体名称（即 SPN）形式的服务绑定信息。</span><span class="sxs-lookup"><span data-stu-id="264f6-127">Service Binding information in the form of a Service Principal Name or SPN.</span></span>  
  
 <span data-ttu-id="264f6-128">服务绑定信息指示客户端想对特定服务终结点验证自己的身份。</span><span class="sxs-lookup"><span data-stu-id="264f6-128">Service Binding information is an indication of a client’s intent to authenticate to a particular service endpoint.</span></span> <span data-ttu-id="264f6-129">它随以下属性从客户端传输到服务器：</span><span class="sxs-lookup"><span data-stu-id="264f6-129">It is communicated from client to server with the following properties:</span></span>

- <span data-ttu-id="264f6-132">SPN 在传输过程中必须受到加密保护，以便中间人攻击不会插入、删除或修改它的值。</span><span class="sxs-lookup"><span data-stu-id="264f6-132">The SPN must be cryptographically protected in transit such that a man-in-the-middle attack cannot insert, remove or modify its value.</span></span>  
  
 <span data-ttu-id="264f6-133">CBT 是外部安全通道（如 TLS）的属性，用于通过验证了客户端的内部通道将其绑定到会话。</span><span class="sxs-lookup"><span data-stu-id="264f6-133">A CBT is a property of the outer secure channel (such as TLS) used to tie (bind) it to a conversation over an inner, client-authenticated channel.</span></span> <span data-ttu-id="264f6-134">CBT 必须具有以下属性（也由 IETF RFC 5056 定义）：</span><span class="sxs-lookup"><span data-stu-id="264f6-134">The CBT must have the following properties (also defined by IETF RFC 5056):</span></span>

- <span data-ttu-id="264f6-139">CBT 在传输过程中必须受到加密完整性保护，以便攻击者不会插入、删除或修改它的值。</span><span class="sxs-lookup"><span data-stu-id="264f6-139">The CBT must be cryptographically integrity protected in transit such that an attacker cannot insert, remove or modify its value.</span></span>
